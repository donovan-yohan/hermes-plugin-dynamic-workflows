# Architecture Stack-Up: hermes-plugin-dynamic-workflows vs. deepagents Dynamic Subagents

*Deterministic workflow scripts on the Hermes runtime (modeled on Claude Code's Workflow tool) vs. model-driven `task`-tool fan-out on LangGraph. All claims below survived adversarial verification; where the original comparison text was refuted, the correction is used.*

---

## 1. TL;DR

The two systems answer the same question — "who decides the fan-out?" — with opposite architectures. hermes-workflows moves the orchestration decision into a frozen, AST-validated, replay-journaled Python script running in a credential-scrubbed subprocess, and then engineers every property around that: bounded concurrency enforced at two layers, allowlist-only context flow across a process boundary, an authoritative token budget with hard abort, call-granular replay, and durable suspend across process death. deepagents leaves the fan-out decision to the model's turn-by-turn tool calls, and gets in exchange a mature, shipped platform: a rich per-agent config surface (tools, middleware, permissions, HITL interrupts), a shared virtual filesystem as a large-artifact channel, battle-tested LangGraph checkpointing, and interactive approve/edit/reject/respond interrupts. deepagents' June-2026 "Dynamic Subagents" feature is tacit validation of the hermes thesis — a code-driven orchestration layer bolted on because prompt-driven coverage fails ("screening 75 of 500 items and calling it done") — but its dynamism is control-flow only: it dispatches a static roster of pre-declared agents, whereas hermes synthesizes agent shapes per call from mid-run data. The honest inversion is operational maturity: hermes's prompt-agent path terminates in a deterministic `StubAgentRunner` and has never been exercised against a live runner, while deepagents launches real agents end-to-end today. Verification also softened two of our sharpest attacks — the QuickJS layer *does* have a 32-call concurrency semaphore and a deterministic snapshot-replay mechanism, and its sandbox limits *are* documented — so the durable design lead is narrower than the raw comparison text claimed, though mid-eval crash recovery and script replay determinism remain genuinely undocumented on their side. Net stack-up: hermes wins on control-flow rigor, isolation, budgeting, and durability design; deepagents wins on definition richness, artifact plumbing, operational maturity, and interactive HITL; the steal list below closes most of the gap on our side cheaply.

## 2. Architecture Side-by-Side

### 2.1 Subagent definition & dynamic creation

**hermes:** No upfront declaration step exists. A script mints an agent per call with `agent(target, input, *, label, phase, schema, model, effort, isolation, context)` (`src/hermes_workflows/vm_guest.py:287-308`); free-form prompts normalize into a frozen `ChildAgentRequest` (`src/hermes_workflows/agents.py:87-117`), model/effort/schema feed the replay fingerprint so differently-shaped calls are distinct (`vm.py:1325-1342`), and schema-invalid output triggers up to `max_schema_retries` re-invocations with the validation error fed back (`vm.py:825-872`). The caveats are real and confirmed: the launch boundary is an injection seam whose only in-repo implementation is `StubAgentRunner` (`agents.py:129`), `ChildAgentRequest` has **no tools/permissions field** — every synthesized child is capability-opaque — and the named `agentType` selector is an open gap (#92, `DESIGN.md:1552`).

**deepagents:** Subagents are declared upfront at `create_deep_agent(subagents=[...])` as TypedDicts with name/description/system_prompt plus tools/model/middleware/interrupt_on/skills/permissions/response_format (`subagents.py:36-124`), compiled into a name→Runnable dict (`subagents.py:588`) that rejects unknown names (`subagents.py:673-675`). **Correction applied:** the original claim that "the docs state there is no dynamic mechanism" was refuted — the [official subagents docs](https://docs.langchain.com/oss/python/deepagents/subagents) have an explicit "Dynamic subagents" section, and "compiled once" is loosened by a per-call recompile path that applies a dynamic `response_format` via the `__deepagents_subagent_response_format` configurable (`_select_subagent`, `subagents.py:640-653`). What holds: new agent *definitions* (names, prompts, toolsets) cannot be created mid-run. The marketed "Dynamic Subagents" feature is confirmed to be dynamic *orchestration* — a QuickJS script calling `task()` over the pre-declared roster ("Dynamic subagents require two things: subagents to dispatch work to, and a code interpreter" — [LangChain blog](https://www.langchain.com/blog/introducing-dynamic-subagents-in-deep-agents)).

**Read:** hermes has the strictly better call-time design (true runtime agent synthesis, shape-aware replay, schema repair); deepagents has the strictly better definition surface and is the only one of the two that actually launches agents. Their dynamism is in control flow; ours is in agent shape — but ours is currently dynamism into a stub.

### 2.2 Orchestration control flow

**hermes:** Plan-in-code is the only path. Scripts pass a two-layer AST gate (`script_validator.py:164-207`) before a subprocess spawns, run with scrubbed env and `PYTHONHASHSEED=0`, and replay via a call-id-keyed cache. Fan-out is structural and confirmed line-by-line: `parallel()`/`pipeline()` admit tasks only while `len(running) < max_parallel` (`vm_guest.py:366, 409`), pipeline is verbatim "bounded per-item stage chains without a cross-item stage barrier" (`vm_guest.py:385`), the first failing branch cancels still-running siblings and re-raises — pipeline failures carrying `item_index`/`stage_index` in `PipelineStageError` (`vm_guest.py:376-380, 400, 419-423, 108-118`) — and the same `max_parallel` independently sizes the parent's `ThreadPoolExecutor` (`vm.py:1603`). A subprocess crash becomes a failed `ScriptRunResult`, never a parent exception (`vm.py:1436-1444`).

**deepagents:** Default fan-out is model-driven and confirmed uncapped: parallelism exists only because the task-tool prompt instructs "use a single message with multiple tool uses" (`subagents.py:290`), and `subagents.py` contains no concurrency primitive whatsoever — execution is delegated to LangGraph's tool layer. `task`/`atask` wrap `subagent.invoke()`/`ainvoke()` in no try/except (`subagents.py:693/721`), and with LangGraph's default `ToolNode` re-raise behavior, one subagent exception propagates uncaught through the parent graph run — confirmed against langgraph and langchain-v1 source. **Correction applied:** the QuickJS layer is better-engineered than the original comparison claimed — `langchain_quickjs/_repl.py` enforces an `asyncio.Semaphore` capping `task()` bridge calls at 32 per interpreter thread (`_repl.py:56/442/630`), implements a deterministic `replay_snapshot_chain` reducer for interpreter state (`_snapshot.py:60`), and catches dispatch exceptions as contained in-script eval errors (`_repl.py:631-640`). The script is still model-authored at request time, and there is no static validation gate before it runs.

**Read:** Still a clear hermes win, on corrected facts. Everything deepagents leaves to the prompt (coverage, caps, barriers, failure containment) hermes enforces in code, and the default deepagents path detonates the parent graph on one branch failure. But the gap on the QuickJS path is caps-and-snapshots vs. our fuller journal — not "engineered vs. nothing." Honest self-criticism stands: our fan-out plan is also typically model-*authored*, just frozen and audited before execution; nested `workflow()` is denied (#91); `parallel`/`pipeline` lack per-call element caps (#94); and the JSON engine's sequential "parallel" gives `max_parallel` two meanings across our two engines — a footgun deepagents avoids by having one semantics.

### 2.3 Context isolation & state

**hermes (all claims confirmed):** Allowlist-only across an OS process boundary. The boot frame carries exactly script/args/limits/budget (`vm.py:1581-1595`); the env is built from scratch, not copied (`_scrubbed_env`, `vm.py:1783-1802`); "The subprocess guest never receives parent chat history or credentials" is verbatim in the `ChildAgentRequest` docstring (`agents.py:91`); unknown option keys raise `CapabilityDenied` at the broker (`vm.py:1297-1322`). Child→parent is a schema-validated flat dict with repair retries; errors are metadata-only (type/message/code/line, no traceback — `vm_guest.py:467-492`). The token budget is authoritative and parent-side: hard abort at ceiling (`vm.py:773-778`, `VMLimits.token_budget` `vm.py:146`), mirrored read-only to the guest on every ret frame (`vm_guest.py:121-154`).

**deepagents (confirmed):** Subtractive denylist in both directions: the child gets the entire parent state minus `{messages, todos, structured_response}` and inbound-only `private_state_keys` (`subagents.py:249-253, 663-664`); the child's mutated non-excluded state silently merges back via `Command(update=...)` plus one ToolMessage (`subagents.py:600-633`). The shared virtual filesystem flows through this channel — a genuine by-reference medium for large artifacts that never transits message context. No token/cost budget exists anywhere in the tree: only call-count caps (`ModelCallLimitMiddleware`/`ToolCallLimitMiddleware`, [middleware docs](https://docs.langchain.com/oss/python/deepagents/middleware)); an adversarial grep confirmed every "budget" hit is context-size management, not spend enforcement.

**Read:** hermes wins quarantine and accounting decisively — deepagents' denylist copy means any new middleware state key flows to children by default unless someone remembers to mark it private, the classic denylist failure mode, and it cannot answer "how many tokens has this fan-out burned" without external tracing. deepagents wins the data plane: hermes has no spill tier (gap #93), so a huge subagent result lands inline in script memory and the replay cache unbounded — our single worst context exposure, and precisely what their filesystem solves.

### 2.4 Durability, resume & operations

**hermes (confirmed):** Every brokered call is journaled by ascending call id into a metadata-only per-run store; `replay_from` identity-checks `script_sha256`/`args_hash` and re-executes with completed deterministic calls served by call id and completed `agent(prompt,...)` results by a v2 semantic fingerprint (`vm.py:463-469, 531-548, 609-645`), failing closed on drift (`vm.py:664-683`). Kanban awaits suspend durably across process death (`vm.py:1046-1093`; `suspended_runs()` `script_store.py:850-873`) against an append-only, fsynced per-card event log (`script_store.py:964-1063`) with idempotent content-addressed card reattach. Per-subagent transcript artifacts persist as JSONL + atomic meta sidecars (`TranscriptRecorder`, `script_store.py:407-573`) — with the verified nuance that these are lifecycle-event records (started/result/error, tokens, durations), not full message-by-message conversations. Cleanup is a first-class finalizer contract (`resources.py:301-447`). Self-documented gaps stand: general partial-run resume and durable side-effect dedup are unsolved (`README.md:475`, `DESIGN.md:1505-1507`), stores are local-file, and the whole story has only run against the stub.

**deepagents (correction applied):** LangGraph checkpointing is per-superstep with production Postgres/SQLite savers and tunable exit/async/sync durability; "Nodes after the checkpoint re-execute, including any LLM calls, API requests" ([checkpointers docs](https://docs.langchain.com/oss/python/langgraph/checkpointers)), with pending writes preserving already-succeeded siblings within a failed superstep. HITL is genuinely richer: `interrupt_on` pauses durably and resumes via `Command(resume=...)` with approve/edit/reject/respond decisions. The interpreter runtime is confirmed beta, **but** the original claim of undocumented sandbox limits was refuted: the [interpreters docs](https://docs.langchain.com/oss/python/deepagents/interpreters) specify a 64 MB default memory limit, 5 s per-eval timeout, no filesystem/network/shell/clock access, and a documented snapshot durability model (interpreter linear memory serialized to graph state at turn boundaries, restored on resume). What remains genuinely undocumented on their side: recovery from a crash *mid-eval* and any replay-determinism guarantee for the orchestration script. Subagent transcripts are not persisted — the middleware's own prompt states "The calling agent only sees your final assistant message" (`subagents.py:246`); intermediate work survives only via LangSmith-style tracing callbacks.

**Read:** The replay unit is the crux. LangGraph checkpoints at node/superstep granularity; a QuickJS script fanning out 300 `task()` calls is one tool call, and while turn-boundary memory snapshots exist, a mid-script death has no documented per-dispatch recovery — exactly the failure hermes journals at per-RPC granularity, so resume cost is proportional to un-run work. But their checkpointing layer is proven in production and ours is a 0.1.0 alpha on local files. Design lead: ours. Operational proof: theirs.

## 3. Where deepagents is genuinely ahead

- **It ships.** Declared subagents launch end-to-end via `create_agent` with their own middleware stacks; hermes's prompt-agent path terminates in `StubAgentRunner` (`agents.py:129`) and durability has only ever been exercised against a deterministic stub.
- **Per-agent definition surface:** tools list, middleware pipeline, filesystem permissions, skills, `interrupt_on`, per-agent model, `response_format` — hermes cannot scope a child's toolset at all (no field exists on `ChildAgentRequest`).
- **Shared virtual filesystem as a by-reference artifact channel** — large work products bypass both the parent context window and the return message. Our gap #93 is the mirror image.
- **Interactive HITL:** approve/**edit**/reject/respond decisions on pending tool calls, resumable across process death via `Command(resume=...)` — richer than our pre-provisioned `approval_ids` plus coarse pause/stop.
- **Mature persistence layer:** production Postgres/SQLite checkpointers, tunable durability modes, pending-writes recovery; ours is alpha and local-only.
- **Three agent kinds:** `CompiledSubAgent` (any prebuilt LangGraph runnable) and `AsyncSubAgent` (remote background runs with start/check/cancel/list) have no hermes equivalent.
- **A working registry:** name→runnable dict plus CLI file-based `agents/{name}/AGENTS.md` with project-over-user override; our `KNOWN_AGENTS` is a demo frozenset whose `hermes.*` wildcard accepts any typo'd id.
- **Provider-native structured output** (`response_format` through `create_agent`), stronger typing than our hand-rolled flat `{field: type}` checker duplicated in two engines (`vm.py:1252`, `runtime.py:630`).
- **One execution semantics.** We have two engines in which `parallel` and `max_parallel` mean different things (real concurrency in the script VM, sequential guard in the JSON runtime).
- **The QuickJS layer is not naive** (post-correction): a 32-per-thread task-dispatch semaphore, deterministic state-snapshot replay reducer, documented 64 MB/5 s sandbox limits, and contained in-script dispatch errors.

## 4. Where hermes-workflows is genuinely ahead

- **True mid-run agent synthesis:** prompt, schema, model, effort, isolation, context chosen per call from script data (`vm_guest.py:287-308`); deepagents freezes every agent shape at build time — its only call-time escape is the `response_format` recompile configurable.
- **Enforced determinism, not aspiration:** AST gate, no import/clock/random/env, `PYTHONHASHSEED=0`, call-id replay cache, fail-closed `replay_mismatch` — vs. a model-authored script with no static gate and no replay-determinism claim.
- **Bounded fan-out by construction:** dual-layer `max_parallel` (guest admission `vm_guest.py:366/409` + parent executor `vm.py:1603`); deepagents' default path has zero caps and coverage-by-prompt.
- **Explicit barrier semantics:** documented no-cross-item-barrier pipeline vs. `phase()` barriers; deepagents has no barrier concept.
- **Failure containment:** catchable `CapabilityError` in-script, deterministic sibling cancellation with index-carrying `PipelineStageError`, subprocess death classified into a failed result — vs. one unguarded `invoke()` exception detonating the parent graph (confirmed against langgraph defaults).
- **Allowlist isolation across a process boundary:** explicit-fields-only contracts both directions, scrubbed env, unknown-key rejection — vs. subtractive full-state copy with silent merge-back.
- **The only token budget in either system:** authoritative, hard-abort, script-visible (`vm.py:773-778`, `vm_guest.py:121-154`).
- **Call-granular durability:** per-RPC journal, fingerprint-served completed agent calls, durable Kanban suspend + idempotent reattach; LangGraph's replay unit is the superstep and re-runs LLM calls past the checkpoint.
- **Built-in transcript artifacts** (JSONL + meta sidecars, `script_store.py:407-573`) with no tracing-SaaS dependency; deepagents discards subagent messages by design.
- **Resource cleanup as a contract:** finalizer policy tiers with a required-failure gate (`resources.py:301-447`); deepagents has nothing comparable.

## 5. Steal list (small → large)

1. **Size-bound agent results.** Apply a `max_result_bytes`-style clip to `agent`/`kanban_agent` results the way `capability()` results are already clipped — enforce in the broker success path in `vm.py` before `_persist_success`, so oversized results can't bloat the replay cache. Closes the sharpest edge of #94/#93 in an afternoon.
2. **Model-legible soft errors for recoverable dispatch failures.** Where a denial is recoverable (unknown agent id, transient runner failure), return a structured catchable error rather than run-terminating — extend the `CapabilityError` taxonomy in `vm.py`/`vm_guest.py:467-492` with a `retryable` flag.
3. **`tools`/permissions allowlist on `ChildAgentRequest`.** Add a `tools` field to the dataclass (`agents.py:87-117`), to `CHILD_AGENT_OPTION_KEYS` (`agents.py:35-37`), thread through `agent()` in `vm_guest.py:287-308` and the broker validation at `vm.py:1297-1322`; the runner seam consumes it when a live runner lands. deepagents' per-agent `tools` + `FilesystemPermission` is the model.
4. **`private_state_keys`-style quarantine spec for the runner seam.** Replace the free-form `context` dict with an explicit host-declared child-visible-keys contract on `ChildAgentRunner` (`agents.py:121`), so the injection seam has a quarantine spec before any live wiring.
5. **JSON-Schema-subset output validator.** Replace the duplicated flat `{field: type}` checkers (`vm.py:1252`, `runtime.py:630`) with one shared validator supporting nested objects/enums, keeping parent-side enforcement and the existing schema-retry loop (`vm.py:825-872`).
6. **Close #92 with the file-based registry UX.** Named agent-type definitions on disk (frontmatter: name/description/model; body = system prompt), project scope over user scope, resolved by the broker at call time — reuse `script_catalog`'s safe-name/root-escape machinery, and auto-register a `general-purpose` default so `agent(prompt)` has defined default semantics.
7. **Truncate-and-spill artifact tier (gap #93).** Results above a threshold get written to the run store (`script_store.py`) and returned to the script as a capped summary + reference handle; the replay cache stores the reference, not the payload. This is the deepagents filesystem idea, adapted to our by-value RPC model.
8. **Durability-mode knob on `ScriptRunStore`.** An `exit/async/sync` fsync policy on journal writes in `script_store.py`, letting operators trade persistence latency against crash-window size, mirroring LangGraph's durability modes.
9. **Pending-writes contract for mid-`parallel()` crashes.** We already have the mechanism (per-call journaling + fingerprint replay); state and test the guarantee that completed sibling branches are cache-served on resume — a fixture in the spirit of `tests/test_subagent_transcripts.py`, documented in `DESIGN.md`.
10. **Interactive interrupt decisions.** Extend the operator control surface (`controls.py`, `CapabilityPolicy.approval_ids`) so a suspended run exposes the pending call and accepts approve/**edit-args**/reject/respond, not just pre-provisioned ids — deepagents' `Command(resume={"decisions": [...]})` is the shape to copy.
11. **Pluggable store backends.** A SQLite (then Postgres) `ScriptRunStore` adapter to close the acknowledged local-only multi-host gap — the largest item, prerequisite to any real fleet deployment.
12. **Async remote-run lifecycle as script globals.** Generalize `kanban_agent`'s blocking/suspending await into a start/check/cancel/list poll surface (the `AsyncSubAgentMiddleware` pattern) for non-Kanban hosts.

## 6. Non-goals

- **Model-driven default fan-out.** Do not add an "agent just calls task() freely" mode. Uncapped, prompt-instructed batching is the exact failure class ("75 of 500 items") the plan-in-code architecture exists to eliminate; deepagents itself had to bolt on a code layer to fix it.
- **Subtractive state copy to children.** Never replace the `ChildAgentRequest` allowlist with a "everything minus a denylist" flow. deepagents' `_EXCLUDED_STATE_KEYS` design means every new state key leaks to children by default — adopting it would forfeit our strongest verified property.
- **In-process isolation.** Do not trade the subprocess boundary for middleware-level isolation to gain speed. The scrubbed env, metadata-only error frames, and crash containment (`vm.py:1436-1444`) all depend on the process boundary.
- **A second (JS) interpreter runtime.** QuickJS solves a problem we don't have — we already possess a validated, replayable script VM. A second language runtime doubles the validation, replay, and sandbox surface for zero capability gain.
- **Silent state merge-back / unbounded return channels.** The `Command`-merge pattern where any key a child mutates lands in the parent is expressive but unauditable; our schema-validated return dict (soon JSON-Schema-subset, steal #5) is the right contract. Adopt the *filesystem-as-artifact-channel* idea (steal #7) explicitly, not implicit state merging.
- **LangGraph/middleware framework coupling.** The zero-dependency posture is load-bearing for the AST-gate + scrubbed-subprocess security story and for auditability. Steal patterns (registry UX, durability modes, interrupt decisions), never the dependency.
- **External-tracing-dependent observability.** deepagents accepts that subagent internals exist only in LangSmith traces. Our local transcript artifacts (`script_store.py:407-573`) are the correct default; deepening them (fuller message capture) beats outsourcing them.
- **Soft-string acceptance of unknown agent ids.** deepagents returns a prose error the model may ignore; our broker's hard rejection of unknown option keys (`vm.py:1297-1322`) is correct for run integrity. Soften only genuinely recoverable dispatch failures (steal #2), never contract violations — and while at it, tighten the `hermes.*` wildcard in `KNOWN_AGENTS` (`agents.py:44-65`), which currently accepts any typo'd id and is our own quiet violation of this principle.
